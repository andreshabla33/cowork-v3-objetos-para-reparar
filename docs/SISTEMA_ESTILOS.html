Sistema de Estilos

Guía completa de cómo están organizados los estilos visuales de Urpeverso, dónde viven y cómo deben usarse correctamente.

Actualizado: Abr 2026 Motor de estilos: Tailwind CSS v3 (JIT) Tema default: light

Resumen general

El sistema de estilos usa Tailwind CSS como motor, con dos capas de tokens TypeScript encima para mantener consistencia visual y soporte multi-tema.

ℹ️
Una sola fuente de verdad

Todos los colores, tipografías, espaciados y efectos deben venir de lib/theme.ts o styles/design-tokens.ts. Ningún componente debe tener colores hardcodeados directamente.

Principio fundamental

Por qué los estilos deben ser globales y no hardcodeados componente por componente.

  • 1
    Mantenimiento centralizado: si el color primario cambia, se modifica en un solo lugar (el token) y afecta toda la app automáticamente. Con hardcode habría que buscar y reemplazar en decenas de archivos.
  • 2
    Temas funcionales: el sistema soporta 4 temas (light, arcade, dark, space). Si un componente usa bg-zinc-900 hardcodeado en vez de t.surface, ese componente ignora el tema activo y siempre se ve oscuro aunque el usuario tenga el tema claro.
  • 3
    Consistencia visual: los tokens garantizan que todos los botones primarios, inputs, bordes y superficies sean visualmente coherentes en toda la aplicación, sin variaciones accidentales.
  • 4
    Legibilidad del código: t.surface comunica intención semántica ("superficie elevada"). bg-zinc-900 solo dice qué color es hoy, no qué rol cumple.
  • 5
    Escalabilidad: agregar un quinto tema o ajustar la paleta de marca requiere tocar solo los archivos de tokens, no rastrear cientos de clases hardcodeadas en componentes.

Capas del sistema

El sistema tiene 5 capas ordenadas de más bajo nivel a más alto nivel de abstracción.

⚙️

Capa 1 — Configuración del motor

Breakpoints, fuentes, keyframes de animaciones. No contiene colores.

tailwind.config.ts
🌐

Capa 2 — CSS global base

@tailwind base/components/utilities. Clases custom: .glass-card, .scrollbar-hide.

styles/tailwind.css
🧱

Capa 3 — Tokens primitivos

Bloques reutilizables del estilo glass/dark: colores, radios, tipografía, spacing, composites.

styles/design-tokens.ts
🎨

Capa 4 — Tokens semánticos (temas)

Superficie, borde, texto, botones, inputs — con variantes por tema. Fuente de verdad principal.

lib/theme.ts
🖥️

Capa 5 — Componentes

Consumen getThemeStyles(theme). Nunca definen colores propios.

components/**/*.tsx

lib/theme.ts — Tokens semánticos

La fuente de verdad principal. Cada token tiene nombre semántico (describe su rol, no su color). Los componentes deben consumir exclusivamente estos tokens.

Constantes de marca exportadas

También se exportan: COLOR_PAGE_BG, COLOR_BORDER, GRADIENT_PRIMARY, GLASS_BASE, GLASS_PANEL, GLASS_PANEL_LIGHT, GLOW_BRAND, GLOW_CYAN. Úsalos cuando necesites los valores de marca sin pasar por el objeto de tema completo.

Temas disponibles

El sistema soporta 4 temas. El usuario puede cambiarlos desde Configuración → General.

☀️
Default ★

light

Azul-slate suave. #F1F5FA de fondo, sky de acento.

🎮

arcade

Negro + verde neón #00ff41. Estética retro terminal.

🌑

dark

Zinc oscuro con glassmorphism. Legacy, mantenido por compatibilidad.

🚀

space

Azul oscuro con acentos cyan. Legacy, mantenido por compatibilidad.

Para agregar un nuevo tema: añadir un nuevo objeto ThemeStyleSet en lib/theme.ts y registrarlo en THEME_STYLES y THEME_LIST. El tipo ThemeType vive en types/index.ts.

Cómo consumir tokens en componentes

El patrón estándar para aplicar estilos correctamente.

Patrón básico con tema

// 1. Obtener tema del store
import { useStore } from '@/store/useStore';
import { getThemeStyles } from '@/lib/theme';

const MiComponente = () => {
  const theme = useStore(s => s.theme);
  const t = getThemeStyles(theme);

  return (
    <div className={t.surface}>
      <h2 className={t.text}>Título</h2>
      <p className={t.textMuted}>Descripción</p>
      <button className={t.btn}>Acción</button>
    </div>
  );
};

Usando tokens primitivos de design-tokens.ts

import { composites, radius, animation } from '@/styles/design-tokens';

// Para elementos 3D o que no cambian con el tema:
<div className={`${composites.cardGlass} ${animation.transition.normal}`}>
  <button className={composites.buttonPrimary}>Guardar</button>
</div>

Combinando ambos

// Usar tema para colores semánticos + design-tokens para layout/efectos
<div className={`
  ${t.surface}
  ${composites.cardGlass}
  ${radius.card}
  ${animation.transition.normal}
`}>

Correcto vs incorrecto

Ejemplos reales del código actual.

❌ Incorrecto — hardcodeado

// No hacer esto nunca:
<div className="bg-[#050508]">

<div className="bg-zinc-900
  border-zinc-800">

<div className="
  from-blue-600
  via-sky-500
  to-cyan-500">

<div className="bg-[#1a1a2e]">
// Colores que ignoran el tema activo

✅ Correcto — usando tokens

// Tokens semánticos:
<div className={t.bg}>

<div className={t.surface}>

<div className={t.btn}>
// Ya incluye el gradiente correcto

<div className={t.globalNav}>
// Cambia automáticamente con el tema

❌ Gradiente duplicado

// Repetido en 20+ componentes:
className="bg-gradient-to-r
  from-blue-600
  via-sky-500
  to-cyan-500"

✅ Constante centralizada

import { GRADIENT_PRIMARY }
  from '@/lib/theme';

className={`bg-gradient-to-r
  ${GRADIENT_PRIMARY}`}

Reglas del equipo

  • 1
    Nunca hardcodear colores hex (#050508, #1a1a2e, etc.) en componentes. Si el color no está en los tokens, agregarlo ahí primero.
  • 2
    Nunca usar clases Tailwind de color directamente (bg-zinc-900, text-slate-400, border-white/5) en componentes. Usar el token semántico equivalente.
  • 3
    El gradiente primario es GRADIENT_PRIMARY exportado desde lib/theme.ts. No repetirlo manualmente en cada componente.
  • 4
    Agregar tokens nuevos a los 4 temas. Si se añade un token a ThemeStyleSet, debe tener valor en light, arcade, dark y space.
  • 5
    Componentes 3D/canvas son la excepción. El espacio 3D tiene su propio sistema visual (Three.js). Pueden usar design-tokens para efectos HUD sin pasar por el tema.
  • 6
    Spacing y radios de design-tokens.ts. Usar radius.card, spacing.gap.lg, etc. en lugar de rounded-2xl o gap-4 directo.

Problemas detectados (deuda técnica)

Inconsistencias actuales que deben corregirse gradualmente.

🚨
index.html — fondo hardcodeado

index.html tiene background: #09090b inline en el <html>. Esto fuerza fondo oscuro aunque el tema activo sea light. Debe alinearse con el tema default o setearse por JS al inicializar.

⚠️
Componentes con hex hardcodeados

Varios componentes usan colores hex literales: InvitationProcessor (bg-[#050508]), MeetingRooms (bg-[#1a1a2e]), múltiples componentes de onboarding con #0a0a0c, #1e2328, #c8aa6e. Pendiente migrar a tokens.

⚠️
design-tokens.ts poco adoptado

El archivo existe pero apenas se importa. La mayoría del codebase sigue usando clases Tailwind directas en lugar de los composites y primitivos definidos ahí.

⚠️
Gradiente primario duplicado

from-blue-600 via-sky-500 to-cyan-500 aparece copiado en +20 componentes. La constante GRADIENT_PRIMARY ya existe en lib/theme.ts pero no se usa.

Qué cambió en los estilos

Historial de las decisiones de arquitectura de estilos más importantes.

Cuándo Qué cambió Impacto
2026 Q1 Migración de CDN de Tailwind a compilación local (PostCSS + Vite) Eliminó dependencia de red en runtime, habilitó JIT tree-shaking, mejoró performance inicial
2026 Q1 Creación de lib/theme.ts con ThemeStyleSet y 4 temas Sistema multi-tema funcional. El workspace shell (nav, sidebar, header) ya lo usa correctamente
2026 Q1 Tema default cambiado de dark a light La experiencia default pasó de zinc oscuro a azul-slate claro (#F1F5FA). dark y space quedan como legacy
2026 Q1 Creación de styles/design-tokens.ts Tokens primitivos para el sistema glassmorphism. Definido pero sub-adoptado en componentes
2026 Q1 Adición de breakpoints 3xl (1920px) y 4xl (2560px) en tailwind.config Soporte para monitores ultrawide (27"–32"+)
2026 Q1 Adición del tema arcade (verde neón / negro) Nuevo tema opt-in con identidad retro. Disponible en Configuración → General
